# Spec Gap Audit D — Pantheon Frontend Build Spec v4 Second-Order Gaps

- 報告 ID：`spec-gap-2026-05-06-D`
- 日期：2026-05-06
- 範圍：v4 規範（含 Pack A/B/C 全部 170 條收斂後）落地至 `src/` 與 `mock BFF` 過程中，發現規範本身仍未明確定義、或定義之間互相矛盾、或缺少最小可實作細節的「次階缺漏」（second-order gaps）。
- 條目數：63（21 High / 28 Medium / 14 Low）
- 狀態：OPEN，待規劃團隊以 Pack D 規範回應。
- 命名規則：依 `.lovable/audits/INDEX.md` 既有約定，序號 `D`。本檔不覆蓋 A / C / C-planner。

---

## 0. 摘要

Pack C 之後，v4 規範解決了 170 條 SA-SD gaps，但實作端在把規範轉成 TypeScript 型別、mock BFF DTO、state machine transition、permission lookup、SSE replay 行為時，仍會遇到「規範說有這個概念，但沒給可實作細節」的情況。本次 Audit D 將這類缺漏分為 9 大主題，共 63 條。其中：

- **5 條 critical blockers**（D05 / D12 / D22 / D26 / D59）若不補規範，無法寫出符合 spec 的對應實作或測試，目前實作端只能用「合理猜測」填補，會在正式後端對接時直接斷裂。
- **21 條 High** 屬於影響正確性 / 安全 / 互通性的核心規範缺口。
- **28 條 Medium** 屬於影響 UX 一致性、QA 可驗收性、跨頁互動行為。
- **14 條 Low** 屬於文案、tokens、工程便利性。

> **重要**：本次盤點是 spec-level 反饋，不是實作 bug 清單。實作端不會自行補規範；補完規範後再開 Pack D 實作工單。

## 1. 方法論

1. 把 Pack C 78 條規範回應，與 `src/lib/v4/`、`src/lib/bff/`、`src/lib/stateMachines/`、`src/mocks/seed.ts` 逐條對照。
2. 遇到任何一處「實作端必須做選擇，但規範沒有給依據」即記錄為 gap。
3. 對矛盾項：以 v4 → v3 → v2 → v1 順序套用，仍無法解的記錄為 gap。
4. 不重複已在 Audit A / C 列出的條目。
5. 嚴重度判準：
   - **High**：阻塞型別、安全、權限、API contract、狀態機正確性。
   - **Medium**：影響 UX 一致性、QA 可驗收性、跨頁互動。
   - **Low**：文案、tokens、工程便利性。

### A. 狀態機與生命週期

#### D01 [High] Strategy 三軸狀態白名單未交叉定義
- 範圍：`src/lib/v4/index.ts`、`StrategyReviewStatus`、`StrategyDeploymentStatus`
- 現況：v4 把 review 收斂到 4 值、deployment 5 值，但沒有定義「lifecycle × review × deployment」三軸組合的合法矩陣。
- 缺漏：例如 `lifecycle=retired` 時 deployment 是否仍可為 `live`？`review=rejected` 時 deployment 是否強制歸 `not_deployed`？
- 建議：補一張 3D truth table（或退化成 2D + constraint list），列出禁止組合與自動聯動規則。
- 對應 Pack C：C002

#### D02 [High] Persona 狀態機 retire 後是否可 reactivate 未定義
- 範圍：`src/lib/stateMachines/`
- 現況：Persona 有 `retired` 終態，但規範沒說是否單向。
- 缺漏：retire → active 是否合法？若可，需要哪個權限與 confirm token？
- 建議：明確標註終態是 absorbing 或可回流，回流時的 audit reason code。
- 對應 Pack C：C006

#### D03 [High] CapitalPool 凍結（freeze）狀態未列入狀態機
- 範圍：`src/lib/stateMachines/`
- 現況：規範文字提到「凍結資金池」操作，但狀態機列舉沒有 `frozen` 狀態。
- 缺漏：是新狀態還是 flag？凍結期間哪些 action 失效？
- 建議：明確選一條路徑（state vs orthogonal flag），補 transition 與 disabledReasonCode。
- 對應 Pack C：C008

#### D04 [High] Incident 與 Deployment rollback 雙寫順序未定義
- 範圍：`src/lib/v4/`、`incidentRollback`
- 現況：incident 可觸發 deployment rollback，但規範沒定義「先建立 incident timeline event 還是先發 rollback command」。
- 缺漏：失敗時的補償順序、idempotencyKey 是否共用。
- 建議：給 sequence diagram + saga compensation 規則。
- 對應 Pack C：C014, C019

#### D05 [High][BLOCKER] 10+ 狀態機缺 timeoutMs / failureState
- 範圍：`src/lib/stateMachines/types.ts`
- 現況：Alert / Incident / CapitalPool / EvolutionRun / Job / Handoff / RoutePolicy / Skill / Memory / Artifact 等狀態機，多數 transition 沒給 `timeoutMs` 與 timeout 後的 `failureState`。
- 缺漏：實作端只能讓 transition 永遠 pending；UI SLA 倒數無基準。
- 建議：每個 async transition 必補三欄：`timeoutMs`、`failureState`、`failureReasonCode`。
- 對應 Pack C：C001, C010

#### D06 [Medium] EvolutionRun pause / resume 是否消耗 quota 未定義
- 範圍：規範 §Evolution
- 現況：暫停期間是否仍計入 fitness budget？
- 建議：明確 quota 計算公式（含暫停時段是否扣除）。
- 對應 Pack C：C015

#### D07 [Medium] Triple-Axis 不變式的觸發 transition 未列舉
- 範圍：v4 §Strategy invariants
- 現況：只說「不變式必須維持」，沒說當哪個 transition 違反時要 reject 還是 auto-correct。
- 建議：每條不變式給「validateOn = [transitionId, ...]」。
- 對應 Pack C：C002

#### D08 [Medium] Job cancel 是否可被 cancel 後 retry 未定義
- 範圍：Job 狀態機
- 缺漏：cancel 後 retry 算新 jobId 還是同 id 重置？
- 建議：明確 retry 策略與 audit chain。
- 對應 Pack C：C020

#### D08b [Low] Memory quarantine 期間是否影響 RAG 檢索未定義
- 對應 Pack C：C006

### B. 權限矩陣與 ActionDescriptor

#### D09 [High] Strategy entity 缺核心 action 列
- 範圍：`src/lib/v4/permissionsMatrix.ts`
- 現況：matrix 沒有 `create_strategy` / `edit_strategy_metadata` / `submit_strategy_review` / `approve_strategy` / `reject_strategy` / `archive_strategy`。
- 建議：補完 Strategy entity 全 action × role 列。
- 對應 Pack C：C013

#### D10 [High] Persona entity 缺核心 action 列
- 範圍：同上
- 缺漏：`create_persona` / `edit_persona_prompt` / `assign_persona_to_channel` / `retire_persona` 全缺。
- 建議：同上。
- 對應 Pack C：C013

#### D11 [High] Channel / Watchlist / Notebook entity 完全未列入
- 範圍：同上
- 建議：補三個 entity 的全部 action 列。
- 對應 Pack C：C013

#### D12 [High][BLOCKER] Role × Capability 對應未明確
- 範圍：`PERMISSION_MATRIX` 的 `capability` 欄
- 現況：每列填了 capability string，但沒有 role → capability bundle 對應表。
- 缺漏：當外部 IdP 回傳的是 capability list（非 role）時，無法 reverse-resolve role。
- 建議：補一張 `RoleCapabilityBundle` 表，並明訂 capability 命名規則（namespace.verb）。
- 對應 Pack C：C013

#### D13 [High] disabledReasonCode 未列舉 enum
- 範圍：`ActionDescriptor.disabledReasonCode`
- 現況：型別是 `string`，沒有受控 enum。
- 缺漏：UI 無法 i18n 對應；不同實作端會發明不同 code。
- 建議：定義 enum，例如 `INSUFFICIENT_ROLE` / `ENV_MISMATCH` / `COOLDOWN_ACTIVE` / `MISSING_TWO_MAN` / `PRECONDITION_FAILED` / `RATE_LIMITED`。
- 對應 Pack C：C013

#### D14 [Medium] requiresEnv 與目前 env 不符時的提示文案未定
- 對應 Pack C：C013

#### D15 [Medium] cooldownSec 倒數的 ground truth（client clock vs server）未定
- 對應 Pack C：C013

#### D16 [Low] capability namespace 是否大小寫敏感未定
- 對應 Pack C：C013

### C. BFF API contract

#### D17 [High] Pagination：未定義 cursor 失效後的 fallback
- 範圍：`src/lib/bff/types.ts`
- 缺漏：cursor 過期或 schema 變更時，server 應回什麼 error code？client 是否可降級為 page-based？
- 建議：定義 `CURSOR_EXPIRED` / `CURSOR_INVALID` 兩個 error code 與 fallback 策略。
- 對應 Pack C：C024

#### D18 [High] Filter operator 未定義
- 範圍：list endpoints
- 現況：規範只說「支援 filter」，沒說語法（`field=value` vs `field[op]=value` vs RSQL）。
- 建議：定義最小 operator set：`eq, in, gt, gte, lt, lte, contains, between`，並給 grammar。
- 對應 Pack C：C024

#### D19 [High] Sort 多欄與 null 排序行為未定義
- 範圍：list endpoints
- 缺漏：`sort=field1,-field2` 是否合法？null 排在前還後？
- 建議：明訂 multi-sort grammar 與 nulls-last 為預設。
- 對應 Pack C：C024

#### D20 [High] 標準 error envelope 欄位 cause 與 details schema 未定
- 範圍：`src/lib/v4/errorEnvelope.ts`
- 現況：`cause?: string`、`details?: Record<string, unknown>` 是自由型。
- 缺漏：跨 service 一致性無法保證。
- 建議：給 `details` 規範（如 validation error 必須是 `{ field, reason, expected }[]`）。
- 對應 Pack C：C027

#### D21 [High] 全平台 error code master list 不存在
- 範圍：跨檔
- 現況：各模組散落 `INSUFFICIENT_ROLE` / `CONFIRM_TOKEN_INVALID` / `IDEMPOTENCY_REPLAY`，沒有單一目錄。
- 建議：建立 `ErrorCode` enum 與 i18nKey 對應表。
- 對應 Pack C：C027

#### D22 [High][BLOCKER] List endpoint 是否回 totalCount 未定
- 範圍：`PaginatedResponse`
- 現況：cursor pagination 通常無 totalCount，但 UI 要顯示 "X of Y"。
- 缺漏：要不要給 `estimatedTotal`？精度保證？
- 建議：定義 optional `estimatedTotal` 與 `totalCountExact` flag。
- 對應 Pack C：C024

#### D23 [Medium] 部分成功（207-style）的 envelope 未定
- 對應 Pack C：C025

#### D24 [Medium] 檔案上傳（attachment）的 multipart 規格未定
- 對應 Pack C：C033

#### D25 [Medium] BFF 速率限制 header（X-RateLimit-*）未列入規範
- 對應 Pack C：C027

#### D26 [High][BLOCKER] SSE channel 每個事件 payload schema 未列
- 範圍：`src/lib/bff/realtime.ts`
- 現況：規範列了 channel name 與「會推什麼類型事件」，但沒給每個事件的具體 payload schema。
- 缺漏：UI 無法寫 type-safe handler，client store reducer 只能 `any`。
- 建議：每個 channel 給 discriminated union，例：`{ type: 'strategy.review.updated', strategyId, reviewStatus, at }`。
- 對應 Pack C：C028

#### D27 [Medium] SSE Last-Event-Id 過期視窗（replay window）秒數未定
- 對應 Pack C：C029

#### D28 [Medium] SSE heartbeat 間隔與超時門檻未明訂
- 對應 Pack C：C029

#### D29 [Medium] resync_required 後的 client 行為（refetch which endpoint）未列
- 對應 Pack C：C029
