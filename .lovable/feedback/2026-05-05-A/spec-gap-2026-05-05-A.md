## 1. Part 1 — Master Blueprint gaps

### G01 [H] Strategy 8 狀態與 Persona/CapitalPool 狀態未在 Part 1 全部展開
- Spec 位置：Part 1 §11 Shared Entity Types（行 1177–1230）  
- 原文摘錄：「Strategy lifecycle: discovered → scaffolded → … → retired」（僅以箭頭列示，無欄位 / 觸發者 / role 對應）  
- 缺漏內容：Part 1 只列出箭頭式狀態流，沒有「該狀態對應哪些欄位才有效、哪些 role 可觸發、哪些 RBAC 動作可呼叫」表格。  
- 跨章矛盾：Part 6 §5.1 列 8 狀態，Part 7 §17.1 也是 8 狀態，**但 Part 8 §7.1 mock 範例出現 `under_review` 與 `paused`，共出現 10 個值**。  
- 目前實作：`src/lib/stateMachines/strategy.ts` 只實作 spec 主表的 8 個。`under_review`、`paused` 在 mock 內被視同 `paper`、`degraded` alias。  
- 影響範圍：StrategyDetail / Strategies list / availableActions 計算 / governance 流程。  
- 建議方向：A) Part 1 §11 補一張「Status × Role × Action」3D 矩陣；B) 把 `under_review`、`paused` 正式納入狀態機；C) 把 Part 8 mock 改回主 8 狀態。  
- 驗收條件：Part 1 / Part 6 / Part 7 / Part 8 對 Strategy 出現的 status enum 完全一致。

### G02 [H] Role 清單與 Action 權限矩陣（Permission Truth Table）完全缺漏
- Spec 位置：Part 1 §1.1（行 55–76）僅列出產品目標；Part 2 §3 列 8 個 role 但無 action 矩陣。  
- 原文摘錄：「Admin / Research Lead / Risk Officer / Capital Manager / Strategy Manager / System Operator / Reviewer / Capability Admin」  
- 缺漏內容：沒有任何「Role × Action」真值表；BFF 應如何回 `availableActions` 沒有可勾稽依據。  
- 跨章矛盾：Part 2 §4.2 要求 BFF 回 `availableActions`，但沒給規則。  
- 目前實作：`src/lib/permissions.ts` 內以「Role → Action 字串集合」自行外推；`permissions.test.ts` 為自訂測試非 spec-driven。  
- 影響範圍：所有 PermissionAwareButton、ActionMenu、HighRiskConfirmationModal。  
- 建議方向：A) Part 1 新增 §11.x「Role × Entity × Action」矩陣（單張 markdown table）；B) 由 BFF spec（Part 6）提供以 fixture 形式涵蓋。  
- 驗收條件：對 Strategy / Persona / CapitalPool / Deployment / Skill / Tool / MCP / Memory 至少 8 種 entity，每個 action 都能 lookup 出准許 role 集。

### G03 [H] High-Risk Action 未列舉完整集合與 confirm token 規格
- Spec 位置：Part 1 §12 High-Risk Action Model（行 1231–1271）  
- 原文摘錄：「Confirmation modal must require: typed entity name, optional memo, confirm token returned by BFF」  
- 缺漏內容：沒列出「哪些 action 是 high-risk」的完整清單；沒定義 confirm token 的取得 endpoint、有效期、refresh 規則。  
- 跨章矛盾：Part 2 §4.3 出現「Promote to live / Rollback / Force kill / Override allocation / Reset memory / Revoke skill / Pause MCP / Approve & deploy / Stop runtime」9 個動作；Part 8 §QA 出現另外 11 個。  
- 目前實作：`src/lib/bff/mutations.ts` 中以 `HIGH_RISK_ACTIONS` 常數 hardcode 約 12 個，confirm token 為 mock 立即回傳。  
- 影響範圍：HighRiskConfirmationModal、Audit memo enforcement、incident kill switch。  
- 建議方向：A) Part 1 §12 補完 Hi-Risk action 全集 + memo mandatory flag；B) Part 6 §3.6 加 `POST /commands/confirm-token`。  
- 驗收條件：每個 hi-risk action 明列：所屬 entity、memo 必填、可觸發 role、confirm token TTL。

### G04 [H] EnvironmentIndicator 未定義環境枚舉與切換規則
- Spec 位置：Part 1 §4.1 Global Top Bar / Environment Indicator（行 320–341）  
- 原文摘錄：「Environment indicator must show current env (dev / staging / prod) and lock destructive actions in non-prod」  
- 缺漏內容：未列舉 env enum 完整集合（demo？sandbox？replay？）；「lock destructive actions in non-prod」與其他段落「only prod allows live」邏輯衝突，未說明何者為主。  
- 目前實作：`src/platform/components/GlobalTopBar.tsx` 寫死 dev/staging/prod 三選；非 prod 不 disable hi-risk。  
- 建議方向：A) Part 1 補 env enum + 對應 action gating 規則；B) 確認 demo 環境是否屬於 staging。  
- 驗收條件：env enum + 每環境允許的 hi-risk action 表。

### G05 [H] BFF availableActions 計算規則未定義
